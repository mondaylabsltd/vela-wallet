# 上架后 90 天:继续全职 / 搁置打工 的决策门槛

> 2026-07-02 预先承诺。目的:在情绪最少的今天定好数字,90 天后照章执行,**不许移动球门**。
> 决策语义:GREEN = 数据证明存在通往 $10K MRR(既定目标)的可信轨迹,继续全职;
> YELLOW = 单一瓶颈可修,延长**一个且仅一个**季度;RED = 转打工,项目降级为受养侧项目。

---

## 〇、先算清这笔账(为什么门槛长这样)

- 单价 $39.99,净收 ≈ $34/单(15% 抽成后,税前)。$10K MRR ≈ **295 单/月** ≈ 68 单/周
- 90 天内不可能到 295 单/月——**所以门槛考的不是绝对收入,是"轨迹 + 漏斗质量 + 可重复性"**:第 3 个月的月收入、月环比增速、以及是否存在一个不靠运气的获客渠道
- relayer 收入 90 天内可忽略(L2 单用户年贡献几美元量级),但**活跃钱包数**是它的先行指标,计入观察
- 现金到账滞后:Apple 按月结算、约次月中下旬打款——**90 天决策看的是确认收入,不是银行余额**,现金流规划另算

**GREEN 轨迹长什么样(倒推示例):** 第 3 个月 $3K + 月环比 ≥30% → 第 6 个月 ≈$6.5K → 第 9 个月 ≈$12K。即 GREEN ≠ 已经养活自己,GREEN = **9 个月内看得见养活自己**。

## 一、计时与有效性规则(先于一切)

1. **计时起点**:双商店都上线且 [distribution-plan](distribution-plan.md) 的 P1 三板斧(Show HN / Product Hunt / 目录)全部**实际执行完**的那一天。发了但扑街 = 有效执行;没发 = 时钟不启动(纪律问题,不是市场信号)
   **执行主体(2026-07-02 决定)**:第一个 90 天由**创始人亲自主导**营销,同事辅助(分工见 [marketing-onboarding.md](marketing-onboarding.md) 模式修正节)——确保对表数据反映市场的真实回答,而非新人学习曲线。窗口内产品冻结为只修 bug
2. **度量前置**:CF Analytics、漏斗周表、商店后台、bundler 新钱包周报必须在上架前就位(见 distribution-plan 度量节)——没有数据的 90 天等于白烧
3. **期间纪律**:价格不动($39.99;区域 PPP 价按计划设好后不动)、不做恐慌性促销、每周 8-10 小时营销节奏不断档。变量越少,90 天后的数据越可信

## 二、三级门槛(第 90 天对表,以第 61-90 天即"第 3 个月"为准,剔除发布尖峰)

### 🟢 GREEN —— 继续全职(四条**同时**满足)

| # | 指标 | 门槛 | 为什么是这个数 |
|---|---|---|---|
| G1 | 第 3 个月确认收入(店销+relayer) | **≥ $2,500**(≈74 单/月,≈17 单/周) | 配合 G2 的增速,9 个月内投影到 $10K MRR |
| G2 | 月环比增速(M2→M3) | **≥ +30%**,且第 3 个月内周销无连续 3 周下滑 | 没有增速的收入是发布余温,不是生意 |
| G3 | 可重复渠道 | 最后 4 周,**非发布事件驱动**的销量 ≥ 10 单/周,且能说出来自哪个渠道 | 证明关掉发布烟花后引擎还在转 |
| G4 | 漏斗底盘 | Web 激活率(创建→入金+首笔)≥ 10% 且 激活→购买 ≥ 5% | 证明 $39.99 模式本身成立,继续投入是放大而非赌博 |

### 🟡 YELLOW —— 延长一个季度(有且仅有一次,附带条件)

触发条件:**G1 在 $1,000–2,500 之间**,且以下诊断表能指出**单一**可修瓶颈:

| 瓶颈层 | 症状 | 修复假设(延长期内验证) | 修不好则 |
|---|---|---|---|
| 曝光 | 90 天落地页 <15k 访问(发布扑街) | 换渠道重打:第二波 HN 角度/播客/本地化市场点火——**营销执行问题,最便宜可重试** | RED |
| 试用→激活 | 大量创建钱包但 <8% 入金 | Onboarding 三步引导/文案信任修复 | RED |
| 激活→购买 | 激活健康但 <2% 买 | **最危险信号**(价格/品类被拒)。唯一预授权动作:执行 pricing-analysis 预设的一次性退阶 $29.99 + Founders 补偿叙事;**只许退这一阶** | RED |

YELLOW 的铁律:延长期开始前**书面写下**瓶颈假设和验证指标;第 180 天二次对表,达不到 GREEN → **自动 RED,无第二次 YELLOW**。

### 🔴 RED —— 搁置,打工养项目(任一条触发且无 YELLOW 资格)

- 第 3 个月收入 **< $1,000** 且无增速(<30 单/月 = 即便乐观复利也在一年内到不了生存线)
- 或:激活→购买 **≈ 0**(>500 个激活用户中购买 <5 单)——付费模式被市场结构性拒绝
- 或:两层以上漏斗同时不及格(不是"单一可修瓶颈",是模式问题)

## 三、中途检查点(不用等 90 天才知道死活)

| 时点 | 检查 | 动作 |
|---|---|---|
| **第 30 天** | 发布烟花后的第一个平常周:周销是否 >0?激活用户 >200? | 若激活 >500 但购买=0 → 提前启动"激活→购买"诊断,**同时开始平行准备简历**(不冲突:准备≠决定) |
| **第 60 天** | M2 收入与漏斗四率成型 | 若已明显 RED 轨迹 → 允许提前对表(提前认输不是违规,提前宣布胜利才是) |
| **第 90 天** | 全量对表,当天出结论 | 结论写进本文档末尾,注明日期 |

## 四、哪些东西**不算数**(防自欺清单)

- 发布日尖峰销量(只看第 61-90 天)
- GitHub stars、粉丝数、媒体报道——除非它们在漏斗表里变成了试用和购买
- "有个大 V 说要推我们"之类的期票
- 累计总收入(掩盖增速归零;看月度切片)
- 用促销制造的第 3 个月冲量(违反 §一 期间纪律,数据作废)

## 五、RED 的正确执行方式(搁置 ≠ 弃养)

钱包托管着用户资金入口,**不可裸弃**。RED 触发后的最低义务:
1. **安全维护底线**:依赖漏洞修复 + 签名路径 bug 修复照做(基础设施是 CF + bundler,月成本低,打工收入可养)
2. **诚实公告**:发一篇 "Vela is now a nights-and-weekends project"——与 honest-alpha 品牌一致,whitepaper 本来就写了 "if Vela disappears" 的自续方案,提前把它变成用户可执行的现实
3. **不下架**:付费 App 下架伤已购用户;保持可购,停止推广
4. **保留复燃条件**:营销引擎文档化齐全(本目录五份文档),同事若继续兼职跑 P2 节奏,项目在低火上保温;某次行业事件带来自然增长时可重评估
